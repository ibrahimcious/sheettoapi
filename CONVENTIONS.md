# CONVENTIONS.md

Conventions observed in this repository. Inconsistencies are flagged explicitly rather than papered over.

---

## 1. Folder Structure

```
src/
  modules/          # Domain logic â€” one sub-folder per feature
    auth/
      auth.api.ts      # Server functions (createServerFn)
      auth.schema.ts   # Zod input schemas
      auth.utils.ts    # Singleton / configuration (auth instance)
    sheets/
      sheets.api.ts
      sheets.schema.ts
  routes/           # TanStack Router file-based routes
    __root.tsx       # Root layout + shell
    index.tsx        # Landing page (/)
    login.tsx
    register.tsx
    dashboard.tsx
    api.auth.$.ts    # API route â€” better-auth catch-all
    api.sheet.$slug.ts  # API route â€” public sheet endpoint
  shared/           # Cross-feature utilities
    lib/
      prisma.ts      # Prisma client singleton
    middleware/
      db.middleware.ts  # TanStack Start middleware
  generated/        # Auto-generated by Prisma â€” never edit manually
    prisma/
  router.tsx        # Router factory + type registration
  routeTree.gen.ts  # Auto-generated by TanStack Router â€” never edit
  styles.css        # Global Tailwind entry point
prisma/
  schema.prisma     # Database schema
  migrations/       # SQL migration files (one folder per migration)
public/             # Static assets
```

**Rule of thumb:** business logic lives in `src/modules/<feature>/`, routing/UI lives in `src/routes/`, cross-cutting infrastructure lives in `src/shared/`.

## 2. File Naming Conventions

### Module files

Use dot-separated `<module>.<role>.ts`:

```
auth.api.ts       # server functions exported for use in routes
auth.schema.ts    # Zod schemas for that module
auth.utils.ts     # singleton or config (e.g., the `auth` instance)
```

### Route files

TanStack Router file-based convention â€” dots become path separators:

| File | Route |
|---|---|
| `index.tsx` | `/` |
| `login.tsx` | `/login` |
| `dashboard.tsx` | `/dashboard` |
| `api.auth.$.ts` | `/api/auth/*` (catch-all) |
| `api.sheet.$slug.ts` | `/api/sheet/:slug` |
| `__root.tsx` | Root layout (not a URL segment) |

### Middleware files

`<name>.middleware.ts` â€” e.g., `db.middleware.ts`.

### Lib files

Plain noun: `prisma.ts`.

## 3. Code Patterns

### Modules (`src/modules/`)

Each module has up to three files with distinct responsibilities:

**`<module>.schema.ts`** â€” Zod schemas only, no logic:
```ts
// sheets.schema.ts
import z from "zod"

export const ConnectSheetSchema = z.object({
  sheetUrl: z.string().min(1, "Sheet URL is required"),
  sheetName: z.string().min(1, "Sheet name is required"),
})

export const DeleteSheetSchema = z.object({
  id: z.string().min(1),
})
```

**`<module>.api.ts`** â€” `createServerFn()` exports, each chained with `.middleware()`, `.inputValidator()`, and `.handler()`:
```ts
// sheets.api.ts
export const connectSheetFn = createServerFn({ method: "POST" })
  .middleware([dbMiddleware])
  .inputValidator(ConnectSheetSchema)
  .handler(async ({ context, data }) => {
    const { db } = context
    // ...
  })
```

**`<module>.utils.ts`** â€” singleton/config that other files import:
```ts
// auth.utils.ts
export const auth = betterAuth({ ... })
```

### Routes (`src/routes/`)

Every route file exports a single `Route` const:
```ts
export const Route = createFileRoute('/dashboard')({
  component: RouteComponent,
  beforeLoad: async () => { ... },
  loader: async () => { ... },
})
```

The component function in route files is named `RouteComponent` by default. Exception: `index.tsx` uses `Home`:
```ts
// index.tsx
function Home() { ... }
export const Route = createFileRoute('/')({ component: Home })
```

**âš ď¸Ź Inconsistency:** `index.tsx` names the component after the page (`Home`), all other routes use the generic `RouteComponent`. Pick one and stick with it.

### Authentication guard pattern

`beforeLoad` fetches the session and redirects if unauthenticated:
```ts
// dashboard.tsx
beforeLoad: async () => {
  const session = await getSessionServerFn()
  if (!session) throw redirect({ to: "/login" })
  return session   // returned value becomes route context
},
```

Access the session in the component via:
```ts
const session = Route.useRouteContext()
```

### Parallel data loading

Use `Promise.all` inside `loader` to fetch independent data:
```ts
loader: async () => {
  const [sheets, userSheets] = await Promise.all([
    getMySheetsFn(),
    getUserSheetsFn()
  ])
  return { sheets, userSheets }
}
```

### Server functions in components

Wrap server functions with `useServerFn` before calling them in event handlers:
```ts
const connectSheet = useServerFn(connectSheetFn)
// then call:
await connectSheet({ data: { sheetUrl, sheetName } })
```

### Middleware

Middleware injects context using `createMiddleware`:
```ts
// db.middleware.ts
export const dbMiddleware = createMiddleware({ type: "function" }).server(
  ({ next }) => next({ context: { db: prisma } })
)
```

Consumed in server functions via `context.db`:
```ts
.handler(async ({ context, data }) => {
  const { db } = context
  return db.sheetConnection.findMany(...)
})
```

### API route handlers

Pure server routes (no React component) use `server.handlers`:
```ts
// api.sheet.$slug.ts
export const Route = createFileRoute("/api/sheet/$slug")({
  server: {
    handlers: {
      GET: async ({ request, params }) => {
        // ...
        return new Response(JSON.stringify(result), {
          status: 200,
          headers: { "Content-Type": "application/json" },
        })
      },
    },
  },
})
```

## 4. Import Ordering and Grouping

Biome is configured with `organizeImports: on`, which auto-sorts imports. The observed grouping in practice:

1. `#/` path-aliased imports (local modules)
2. `@tanstack/*` and other npm packages
3. `react`

Example from `dashboard.tsx`:
```ts
import { getSessionServerFn, logoutServerFn } from '#/modules/auth/auth.api'
import { createFileRoute, redirect, useRouter } from '@tanstack/react-router'
import { useServerFn } from '@tanstack/react-start'
import { connectSheetFn, getMySheetsFn, deleteSheetFn, getUserSheetsFn } from '#/modules/sheets/sheets.api'
import { useState } from 'react'
```

**âš ď¸Ź Inconsistency:** The ordering is not uniform across all files. `auth.api.ts` puts `@tanstack/*` imports before local `./` imports. Biome's `organizeImports` is enabled â€” run `biome check --write` to enforce consistency.

**Path alias in use:** Always use `#/*` (maps to `./src/*`). The `@/*` alias is defined in `tsconfig.json` but not actually used anywhere in source.

## 5. Error Handling Patterns

### Server functions â€” authorization errors

Throw a plain `Error` directly; no custom error class:
```ts
const session = await auth.api.getSession({ headers })
if (!session) throw new Error("Unauthorized")
```

### Server functions â€” redirect after action

Use `throw redirect(...)` (TanStack Router convention â€” redirect is thrown, not returned):
```ts
throw redirect({ to: "/dashboard" })
throw redirect({ href: response.url })   // for external URLs
```

### API route handlers â€” HTTP error responses

Return structured JSON with explicit status codes:
```ts
if (!sheet) {
  return new Response(
    JSON.stringify({ error: "Endpoint not found" }),
    { status: 404, headers: { "Content-Type": "application/json" } }
  )
}
if (!apiKey || apiKey !== sheet.apiKey) {
  return new Response(
    JSON.stringify({ error: "Invalid API key" }),
    { status: 401, headers: { "Content-Type": "application/json" } }
  )
}
```

### Client-side â€” event handlers

`try/catch` with local error state:
```ts
const [error, setError] = useState('')
// ...
try {
  await connectSheet({ data: { sheetUrl, sheetName } })
} catch (e) {
  setError('Failed to connect sheet. Please try again.')
} finally {
  setIsConnecting(false)
}
```

Error is rendered inline near the action:
```tsx
{error && <p className='text-red-500 text-sm'>{error}</p>}
```

There is no centralized error boundary or global error handler.

## 6. Feature Organization â€” Adding a New Module

Pattern derived from observing `auth` and `sheets`:

1. Create `src/modules/<feature>/`
2. Add `<feature>.schema.ts` with Zod schemas for all inputs
3. Add `<feature>.api.ts` with `createServerFn()` exports, using `.middleware([dbMiddleware])` for DB access
4. Add `<feature>.utils.ts` if there is a singleton or complex config
5. Import server functions into route files via `#/modules/<feature>/<feature>.api`
6. Wrap with `useServerFn()` in the component before calling

For a public API endpoint (no React component):
1. Add `src/routes/api.<feature>.$param.ts`
2. Define `server.handlers` with `GET`/`POST` etc.
3. Return `new Response(JSON.stringify(...), { status, headers })` directly

## 7. Naming Conventions

### Variables and functions

camelCase throughout:
```ts
const sheetId = extractSheetId(data.sheetUrl)
const slug = generateSlug(data.sheetName)
async function handleConnect() { ... }
async function handleDelete(id: string) { ... }
```

### Environment variable constants

SCREAMING_SNAKE_CASE at module level:
```ts
const GOOGLE_API_KEY = process.env.GOOGLE_API_KEY
```

**âš ď¸Ź Inconsistency:** Singletons (`auth`, `prisma`) use plain camelCase, but `GOOGLE_API_KEY` uses SCREAMING_SNAKE_CASE. No consistent rule for "constant vs singleton" distinction.

### React components

PascalCase:
```ts
function RouteComponent() { ... }
function RootDocument({ children }: { children: React.ReactNode }) { ... }
function Home() { ... }
```

### Zod schemas

PascalCase + `Schema` suffix:
```ts
export const RegisterUserSchema = z.object({ ... })
export const ConnectSheetSchema = z.object({ ... })
export const DeleteSheetSchema  = z.object({ ... })
```

### Server functions

camelCase + `Fn` suffix. **âš ď¸Ź Inconsistency:** `auth.api.ts` uses `ServerFn`, `sheets.api.ts` uses `Fn`:

```ts
// auth.api.ts â€” uses ServerFn
export const loginServerFn = createServerFn()...
export const getSessionServerFn = createServerFn()...

// sheets.api.ts â€” uses Fn
export const connectSheetFn = createServerFn()...
export const getMySheetsFn = createServerFn()...
```

Pick one suffix and apply uniformly. `Fn` is shorter; `ServerFn` is more explicit.

### Client-side server function handles

When a `useServerFn` wrapper is assigned, use the same name without suffix, or append `Handler`:
```ts
// login.tsx
const loginWithGoogle = useServerFn(loginWithGoogleServerFn)

// register.tsx
const registerServerFnHandler = useServerFn(registerServerFn)

// dashboard.tsx
const logoutServerFnHandler = useServerFn(logoutServerFn)
const connectSheet = useServerFn(connectSheetFn)
```

**âš ď¸Ź Inconsistency:** Some use `Handler` suffix (`logoutServerFnHandler`, `registerServerFnHandler`), some don't (`loginWithGoogle`, `connectSheet`, `deleteSheet`). No consistent rule.

### Route constant

Always `Route` (required by TanStack Router):
```ts
export const Route = createFileRoute('/dashboard')({ ... })
```

### Middleware

camelCase + `Middleware` suffix:
```ts
export const dbMiddleware = createMiddleware(...)
```

### Prisma models

PascalCase (`SheetConnection`, `User`, `Session`). DB field names are camelCase (`userId`, `sheetId`, `apiKey`).

## 8. Comment and Documentation Style

Comments are used selectively but heavily in data-heavy files. Two distinct styles:

### Block comments above functions (used in `sheets.api.ts`, `api.sheet.$slug.ts`)

Single-line `//` before each exported function, explaining its purpose:
```ts
// Connect a Google Sheet â€” saves a new SheetConnection to DB
// Extracts the sheet ID from the URL, generates a slug and API key
export const connectSheetFn = createServerFn({ method: "POST" })
```

Inline `//` comments explaining non-obvious logic:
```ts
// q: filter by Google Sheets mime type only
// pageSize: limit to 10 results
// orderBy: newest modified first
// fields: only return id and name to keep response small
const response = await fetch(
  "https://www.googleapis.com/drive/v3/files?q=mimeType=..."
)
```

### JSX comments inside components (used in `dashboard.tsx`)

`{/* ... */}` to label sections:
```tsx
{/* Header with user info and logout */}
{/* Section 1: Connect a new sheet */}
{/* List of user's Google Sheets fetched from Drive API */}
{/* Empty state */}
```

### No comments

`auth.api.ts` and route files for simple pages (`login.tsx`, `index.tsx`) have no comments at all.

### âš ď¸Ź Bug: JSX comment syntax in a `.ts` file

`auth.schema.ts` line 13 uses JSX comment syntax inside a plain TypeScript file:
```ts
{/* connectSheetSchema validates the user's input... */}
```
This is invalid â€” `{/* */}` is only valid inside JSX expressions. Use `//` or `/* */` in `.ts` files.

## Summary of Inconsistencies

| Area | Inconsistency |
|---|---|
| Server function suffix | `auth.api.ts` uses `ServerFn`, `sheets.api.ts` uses `Fn` |
| Client handle naming | Some add `Handler` suffix after `useServerFn`, some don't |
| Route component naming | `index.tsx` uses `Home`, all others use `RouteComponent` |
| Import ordering | Biome organizes imports but grouping is not consistent across files |
| Constants naming | `GOOGLE_API_KEY` is SCREAMING_SNAKE_CASE, singletons (`auth`, `prisma`) are camelCase |
| Comment density | Heavy in `sheets.api.ts` and `api.sheet.$slug.ts`, absent in `auth.api.ts` |
| JSX comment in `.ts` file | `auth.schema.ts` uses `{/* */}` which is invalid outside `.tsx` |
| `@/*` alias | Defined in `tsconfig.json` but never used; `#/*` is used exclusively |
